**Release Date**: Q2 2025 (Planned)
**Status**: Design Phase

## ğŸ¯ Executive Summary

Wareflow EMS v2.0.0 transforms from a single desktop application into a **application generator** (bootstrapper) that creates customized employee management systems for logistics companies.

### Key Changes

- **Product Pivot**: From "single HR app" to "HR app generator"
- **Command Change**: `employee-manager` â†’ `wms`
- **New Command**: `wms init <company-name>` creates a complete customized app
- **Distribution**: Pre-compiled CustomTkinter executable included
- **Configuration**: Human-friendly YAML files instead of JSON
- **Target Users**: HR consultants, system integrators, IT service providers

### Business Value

Wareflow EMS becomes a **B2B SaaS-on-premise product**:
- Sell to HR consultants who deploy to multiple clients
- Generate customized apps for each company
- Simplified deployment (no Python required for end users)
- Per-company isolated databases
- Automatic updates via regeneration

### YAML Schema Validation

```python
# src/bootstrapper/validators.py

from yaml import safe_load
from jsonschema import validate

CONFIG_SCHEMA = {
    "type": "object",
    "properties": {
        "alerts": {"type": "object"},
        "organization": {"type": "object"},
        "database": {"type": "object"},
        "interface": {"type": "object"},
        "business": {"type": "object"},
        "advanced": {"type": "object"}
    },
    "required": ["alerts", "organization", "database"]
}

def validate_config(config_path: str) -> dict:
    """Validate config.yaml against schema."""
    with open(config_path) as f:
        config = safe_load(f)

    validate(instance=config, schema=CONFIG_SCHEMA)
    return config
```

## ğŸ¯ Use Cases

### Use Case 1: HR Consultant

**Scenario**: Consultant manages HR for 5 different logistics companies

**Workflow**:
1. **Setup** (one-time)
   ```bash
   # Install wareflow-ems
   wms init "ConsultingHR" --template enterprise
   ```

2. **Deploy to clients**
   ```bash
   # For Client A
   wms init "Client A" --output "C:/apps/client-a"

   wms init "Client B" --output "C:/apps/client-b"
   ```

3. **Maintain apps**
   ```bash
   # Update all apps
   for app in C:/apps/*/; do
     cd $app
     wms update
   done
   ```

**Scenario**: MSP manages IT for multiple companies

**Workflow**:
1. **Prepare environment**
   - Install Wareflow EMS on server
   - Create shared folder: `//server/apps/`

2. **Deploy per company**
   - `wms init "Company A" --output "//server/apps/company-a"`
   - Share folder with employees
   - Create shortcut to `\\server\apps\company-a\wms.exe`

3. **Centralized updates**
   - Update `wms.exe` in shared location
   - All apps auto-update

### Use Case 3: Single Company

**Scenario**: Single warehouse operation

**Workflow**:
1. Download `wareflow-ems-installer.exe`
2. Run installer
3. Config wizard launches
4. Choose "Simple Mode"
5. App installed and configured
6. Desktop shortcut created

## ğŸ“ Notes

### Why Not Electron/Web?

**Question**: Why not build a web app instead of desktop?

**Answer**:
- **Offline-first**: Warehouses often have poor internet
- **Performance**: Native GUI faster than web
- **Simplicity**: No web server to maintain
- **Security**: No network attack surface
- **Regulatory**: Data stays on-premise (GDPR friendly)

### Why YAML Not TOML?

**Question**: Why not TOML instead of YAML?

**Answer**:
- **Comments**: YAML supports inline comments
- **Nested structures**: YAML handles deep nesting better
- **Industry standard**: K8s, Ansible use YAML
- **More expressive**: YAML has more features (anchors, aliases)
